# Release process

**Audience:** new NebulaDNS developers, release engineers, and AI coding
agents who need to cut, document, verify, or roll back a release.

**TL;DR:**

```bash
# 1. From a clean green `main`, pick a version (SemVer, v-prefixed)
./xtask/bumpversion.sh v0.2.0      # or edit Cargo.toml + Cargo.lock by hand

# 2. Open a PR titled "release: v0.2.0", get it reviewed + merged

# 3. Tag the merge commit and push
git checkout main && git pull
git tag -a v0.2.0 -m "v0.2.0"
git push origin v0.2.0

# 4. Watch the `release` workflow on GitHub Actions.
#    When it's green, the Release page is live with binaries + checksums.
```

All other detail is below. If you are an agent cutting a release on a human's
behalf, read §§ 1 and 3 before pushing a tag — getting the version wrong or
tagging the wrong SHA is the only way to make this process painful to undo.

---

## 1. Versioning

NebulaDNS follows [Semantic Versioning 2.0.0](https://semver.org/) once we cut
v1.0.0. Until then, we are in the `0.y.z` pre-release plane:

| pre-1.0 change | bumps | example                |
| -------------- | ----- | ---------------------- |
| breaking       | `y`   | `0.1.x` → `0.2.0`      |
| feature or fix | `z`   | `0.1.0` → `0.1.1`      |

Milestone pre-releases use the `-mN.pre` suffix and are **pre-release** on
GitHub (they don't show as the "Latest" release):

- `v0.1.0-m0`, `v0.2.0-m3.pre`, `v0.3.0-rc1`

Any tag whose version contains a `-` after the numbers is treated as a
pre-release by the `release` workflow.

### What's in a version

Each release is identified by a **single annotated git tag**, matching
`v*.*.*` or `v*.*.*-*`, which points at the exact commit that was built. The
tag is the source of truth — the tarball names, the `nebula_build_info`
metric label, the Ansible deploy's `nebuladns_version`, and the Release page
all derive from it.

## 2. Who fires the release

| Action                             | Who                              |
| ---------------------------------- | -------------------------------- |
| Write the release notes (draft)    | PR authors (via labels)          |
| Approve the release PR             | A code owner (see `CODEOWNERS`)  |
| Push the tag                       | The maintainer merging the PR   |
| Publish the Release                | `release.yml` workflow, automated |
| Deploy to staging                  | `deploy.yml` (tag push)          |
| Promote staging → production       | `deploy.yml` (manual dispatch)   |

Tag pushes trigger **both** `release.yml` (publish artifacts) and `deploy.yml`
(deploy to production). If you only want to cut artifacts without a deploy —
for example to publish a pre-release — push a `-*` pre-release tag; the
production deploy job can be skipped by maintainers via environment
protection rules.

## 3. Cutting a release, step by step

### 3.1 Verify `main` is green

```bash
git checkout main && git pull
gh run list --branch main --limit 5
```

The latest `ci` run on the SHA you're about to tag MUST be green. If it's not,
fix the breakage first — **never** tag a red commit. The `release` workflow
also re-runs the test gate against the tagged SHA as a second line of
defence, but you should catch it earlier.

### 3.2 Update the version in the workspace

The canonical version lives in the `[workspace.package]` table of the root
`Cargo.toml`. Update it, then refresh `Cargo.lock`:

```bash
# Edit Cargo.toml — change:
#   [workspace.package]
#   version = "0.1.0-m0"
# to the new version, e.g. "0.2.0".

cargo update --workspace
cargo build --workspace        # sanity check
```

Open a PR titled `release: vX.Y.Z`, get it reviewed, merge it.

### 3.3 Write the release notes by labeling PRs

Release notes are generated by GitHub's native
[auto-generated release notes](https://docs.github.com/en/repositories/releasing-projects-on-github/automatically-generated-release-notes).
The categories — Features, Bug fixes, Security, Docs, etc. — are defined in
[`.github/release.yml`](.github/release.yml) and driven by PR labels.

**Please label your PRs.** An unlabeled PR lands in "Other changes", which
is fine as a fallback but uninformative in the changelog. The label → section
mapping is:

| PR label                | Release-notes section         |
| ----------------------- | ----------------------------- |
| `breaking`              | ⚠️  Breaking changes           |
| `feature`, `enhancement`| 🚀 Features                   |
| `bug`, `fix`            | 🐛 Bug fixes                  |
| `security`              | 🔐 Security                   |
| `ops`, `deploy`, `infra`| ⚙️  Operations / deploy       |
| `metrics`, `observability` | 📊 Observability           |
| `docs`                  | 📚 Documentation              |
| `tests`, `ci`           | 🧪 Tests / CI                 |
| `chore`, `refactor`     | 🧹 Chore / refactor           |
| `skip-changelog`        | (excluded entirely)           |

You do not need to hand-write a changelog. If there's additional context that
belongs in the release — migration notes, security advisories, credit for
reporters — add it as a comment on the release after it's published, or edit
the generated notes directly.

### 3.4 Tag and push

From a clean `main` that includes the version bump commit:

```bash
git checkout main && git pull
git log --oneline -1                  # sanity: is this the commit I think it is?
git tag -a v0.2.0 -m "v0.2.0"
git push origin v0.2.0
```

Use **annotated** tags (`-a`) so the tag itself has a message and author.
`softprops/action-gh-release` and `git describe` both treat annotated tags as
first-class.

### 3.5 Watch the workflow

```bash
gh run watch --exit-status
gh release view v0.2.0 --web        # open the published Release
```

The [`release` workflow](.github/workflows/release.yml) runs:

1. `test-gate` — fmt + clippy + tests against the tagged SHA.
2. `build` — four targets in parallel:
   - `x86_64-unknown-linux-musl`
   - `aarch64-unknown-linux-musl`
   - `x86_64-apple-darwin`
   - `aarch64-apple-darwin`
3. `publish` — combines artifacts, writes `SHA256SUMS`, creates the Release
   with auto-generated notes.

If `build` smoke-test fails, the release does not publish.

### 3.6 Verify the release is actually usable

Before you announce it, download and try it:

```bash
VERSION=v0.2.0
TARGET=x86_64-unknown-linux-musl
gh release download "$VERSION" \
  --pattern "nebuladns-${VERSION}-${TARGET}.tar.gz" \
  --pattern "nebuladns-${VERSION}-${TARGET}.tar.gz.sha256"

shasum -a 256 -c "nebuladns-${VERSION}-${TARGET}.tar.gz.sha256"
tar -xzf "nebuladns-${VERSION}-${TARGET}.tar.gz"
./nebuladns-${VERSION}-${TARGET}/nebuladns --print-default-config | head
```

For a full smoke, use the `deploy/docker/compose` rig with the release tag.

## 4. Release artifacts

Every release publishes the following to the GitHub Release page:

| File                                                      | What it is                                             |
| --------------------------------------------------------- | ------------------------------------------------------ |
| `nebuladns-vX.Y.Z-x86_64-unknown-linux-musl.tar.gz`        | Statically-linked Linux binaries (`nebuladns`, `nebulactl`) + `README`, `LICENSE`, `RELEASE.md` |
| `nebuladns-vX.Y.Z-aarch64-unknown-linux-musl.tar.gz`       | Same, ARM64                                            |
| `nebuladns-vX.Y.Z-x86_64-apple-darwin.tar.gz`              | macOS (Intel)                                          |
| `nebuladns-vX.Y.Z-aarch64-apple-darwin.tar.gz`             | macOS (Apple silicon)                                  |
| `<archive>.sha256`                                         | Per-archive SHA-256 (BSD format)                       |
| `SHA256SUMS`                                               | Concatenated per-archive SHA-256 lines                 |

**Not yet published** (roadmap): container images (ghcr.io), Helm chart OCI
artifact, `cosign`-signed attestations. Tracked in `PROJECT_PROMPT.md §15`.

The Ansible deploy pipeline (`deploy.yml`) pulls its binaries from the
`upload-artifact` cache of its own build job, not from the Release — so a
failed publish step does not block deploy, and a hand-cut release does not
automatically deploy.

## 5. Verifying a release you downloaded

Until we sign releases with cosign, the only integrity check is the
SHA-256 sum. To verify:

```bash
VERSION=v0.2.0
TARGET=x86_64-unknown-linux-musl
gh release download "$VERSION" \
  --pattern "nebuladns-${VERSION}-${TARGET}.tar.gz" \
  --pattern "nebuladns-${VERSION}-${TARGET}.tar.gz.sha256"

shasum -a 256 -c "nebuladns-${VERSION}-${TARGET}.tar.gz.sha256"
#   nebuladns-v0.2.0-x86_64-unknown-linux-musl.tar.gz: OK
```

Or verify all assets at once:

```bash
gh release download v0.2.0 --pattern 'SHA256SUMS' --pattern '*.tar.gz'
shasum -a 256 -c SHA256SUMS
```

## 6. Rolling back

A release, once published, is immutable. "Rolling back" means **cutting a new
release** that reverts the offending changes. Do not rewrite or delete tags;
downstream consumers and the `deploy.yml` job treat the tag history as
authoritative.

### 6.1 Revert the offending PRs

```bash
git checkout main && git pull
git revert <bad-sha-1> <bad-sha-2> ...
# or: cherry-pick forward the good state
```

Open a PR titled `release: vX.Y.Z+1 (rollback of vX.Y.Z)`, get it reviewed
and merged, then cut a new release following § 3.

### 6.2 If a release page itself is wrong (typo, missing asset)

Use `gh release edit` — the **tag** stays put, only the release page text or
asset list changes.

```bash
gh release edit v0.2.0 --notes-file updated-notes.md
gh release upload v0.2.0 new-asset.tar.gz
```

### 6.3 In-place deploy rollback

That's a separate topic owned by `deploy/ansible/rollback.yml`. See
[`CONTRIBUTING.md`](CONTRIBUTING.md) and the runbooks in `docs/runbooks/`.

## 7. For AI coding agents

If you are an agent being asked to cut a release on a developer's behalf,
follow this checklist strictly:

1. **Ask the user for the version number** (`vX.Y.Z`) — do not guess. Confirm
   whether it's a regular release or a pre-release.
2. **Confirm the target branch and SHA.** Run `git log --oneline -5 main` and
   show the user the commit you would tag. Do not proceed until they confirm.
3. **Never** force-push a tag, delete a tag, or amend a published release's
   tag. "Rolling back" means cutting a new version (§ 6).
4. **Never** skip the test gate with `--no-verify` or by disabling CI. If the
   test gate is red, stop and surface the failure to the user.
5. After pushing the tag, report back the URL:
   `https://github.com/bwalia/nebuladns/releases/tag/vX.Y.Z`.
6. Do not announce or cross-post the release. That is a human decision.

Reference files the agent should read before acting:

- This file (`RELEASE.md`) — the process
- `.github/workflows/release.yml` — what actually runs
- `.github/release.yml` — how the changelog is grouped
- `Cargo.toml` — where the version lives
- `CONTRIBUTING.md` — commit conventions, PR expectations
